# Семестровый проект: Angular Application

## Цель

Показать уровень владения материалами второго полугодия (Angular):

- архитектурное мышление в контексте фреймворка;
- компонентная разработка с использованием Angular;
- управление состоянием (RxJS, Signals, NgRx/Signal Store);
- работа с API, авторизация, guards, interceptors;
- тестирование (unit + component);
- CI/CD, деплой, производительность.

Проект выполняется на Angular с использованием mock-сервера.

---

## Пререквизиты

- Успешное завершение первого полугодия (или эквивалентные знания);
- Базовое понимание TypeScript, HTML, CSS;
- Опыт работы с Git и CLI.

---

## Сроки реализации

**Общая длительность:** 2 месяца (начало апреля — конец мая)

### Обязательные дедлайны

| Этап | Срок | Что должно быть готово |
|------|------|------------------------|
| основа приложения | До конца апрель | базовая структура проекта, компоненты, формы, routing |
| рабочий прототип | До конца мая | API integration, state management, auth flow, тесты, CI/CD, деплой |
| завершение разработки | За начало июня | финальная полировка |

---

## Стек и ограничения

- **Фреймворк:** Angular 21 (обязательно);
- **Язык:** TypeScript;
- **UI-библиотека:** Taiga UI 4 (обязательно);
- **State Management:** NgRx или Signal Store;
- **API:** Mock-сервер (обязательно);
- **Хранение данных:** localStorage для токенов и настроек;
- **Тестирование:** Unit-тесты (Jest), component tests (Playwright);
- **Сборка и деплой:** Angular CLI build, GitLab Pages / Firebase / Vercel;
- **Качество кода:** ESLint, Prettier, Stylelint.

---

## Работа с Git и репозиторием

- Основная ветка: `main` или `master`;
- Для проекта создаётся отдельная рабочая ветка;
- Работа ведётся через небольшие отдельные ветки и Merge Request-ы;
- История разработки должна быть поэтапной;
- Сообщения коммитов — по Conventional Commits.

## Оценивание

Максимум за проект — 10 баллов:

| Компонент | Макс. баллы | % от итога |
|-----------|-------------|------------|

Отдельно применяются штрафы (см. ниже).

## 1. Предварительный анализ (до 2 баллов)

### 1.1. План анализа и разработки + риски — до 1 балла

**Файл:** `docs/plan.md`

Содержание:
- цель проекта и краткое описание продукта;
- 2-4 основных пользовательских сценария;
- этапы разработки с привязкой к дедлайнам;
- 3-5 ключевых рисков и способы их снижения.

---

### 1.2. UX-концепция + дизайн-система — до 1 балла

**Файл:** `docs/ux.md`

Содержание:
- целевая аудитория / персоны (1-2 типа пользователей);
- для каждой персоны — цели и ожидания;
- ключевые сценарии в формате user story;
- дизайн-система на базе Taiga UI 4;
- приоритеты: что на первом экране, что вторично.

**Figma (прототип):**
- Минимум 3-4 экрана/состояния;
- Ссылка в `docs/ux.md`.

---

## 2. Реализация (до 5 баллов)

### 2.1. Функциональные требования — до 2 баллов

Требуется реализовать:
- главные пользовательские сценарии из анализа;
- CRUD основных сущностей на mock-данных;
- минимум одна полноценная форма с валидацией;
- интеграция с mock-API;
- обработка loading/error состояний;
- поиск и фильтрация данных;
- сортировка (минимум 1 вариант);
- вычисляемые показатели (прогресс, статистика);
- авторизация:
  - login flow (форма, валидация);
  - хранение токена (localStorage);
  - logout.

---

### 2.2. Архитектура Angular — до 2 баллов

Требуется:
- структура проекта с разделением на features / shared / core;
- lazy loading для маршрутов;
- компоненты с @Input / @Output;
- OnPush change detection где уместно;
- бизнес-логика вынесена в сервисы;
- state management (NgRx или Signal Store);
- guards для защищённых маршрутов;
- interceptors для токенов и обработки ошибок;
- строгая типизация (избегать any).

---

### 2.3. Нефункциональные требования — до 1 балла

Требуется:
- адаптивная вёрстка (desktop / tablet / phone);
- Taiga UI 4 как основная UI-библиотека;
- доступность (a11y): семантический HTML, aria-атрибуты;
- производительность: lazy loading, OnPush, trackBy;
- bundle size < 1MB (production build).

---

## 3. Тестирование (до 1.5 баллов)

### 3.1. Unit-тесты — до 0.5 балла

- инструмент: Jest;
- покрытие: сервисы, компоненты (логика), pipes, guards;
- минимум: 10-15 тестов на ключевую логику.

---

### 3.2. Component tests — до 1 балла

- инструмент: Playwright;
- минимум: 2-3 сценария (login, CRUD, фильтрация).

---

## 4. CI/CD, деплой, линтеры (до 1.5 баллов)

### 4.1. Линтеры и форматирование — до 0.5 балла

Требуется настроить ESLint, Stylelint, Prettier.

---

### 4.2. CI/CD и деплой — до 1 балла

Требуется:
- CI pipeline с этапами: lint, test, build, deploy;
- деплой на GitLab Pages / Firebase / Vercel;
- рабочий публичный URL (ссылка в README.md);
- Lighthouse (mobile): показатели >= 80 по SEO и Best Practices.

---

## Обязательные технические требования

- Angular 21;
- Taiga UI 4;
- State management (NgRx или Signal Store);
- авторизация с токенами и guards;
- interceptors для токенов и обработки ошибок;
- тесты: unit на Jest (10-15) + component на Playwright (2-3 сценария);
- CI/CD pipeline с lint, test, build, deploy;
- деплой на GitLab Pages / Firebase / Vercel;
- Lighthouse >= 80 (SEO, Best Practices).

---

## Штрафы (по -0.5 балла)

Могут суммироваться:

- названия коммитов не соответствуют Conventional Commits (системно);
- отсутствие типизации в публичных API без обоснования;
- Lighthouse < 80% по SEO или Best Practices без попыток исправления;
- отсутствует деплой (нет рабочего публичного URL);
- отсутствует README (описание, структура, инструкция по запуску);
- Taiga UI используется формально (менее 5 компонентов).

---

## За что автоматически 0 баллов

- явная копия чужой работы (плагиат);
- непонимание собственной архитектуры и решений;
- отсутствие истории изменений (1 коммит);
- отсутствие авторизации;
- отсутствие state management (NgRx/Signal Store).

При обнулении проекта обязательна пересдача с новой темой.
