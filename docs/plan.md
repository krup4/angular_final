# План разработки

## Цель проекта

Subscription Manager помогает пользователю контролировать подписки и регулярные расходы: видеть активные сервисы, стоимость в месяц, ближайшие списания, уведомления перед оплатой и превышение лимитов по категориям.

Проект закрывает бизнес-требования темы 16 и критерии Angular-проекта: CRUD, mock API, авторизация, разделение данных по пользователям, state management, guards/interceptors, тесты, CI/CD и адаптивный UI на Taiga UI.

## Пользовательские сценарии

1. Пользователь входит в аккаунт и видит только свои подписки.
2. Пользователь добавляет или редактирует подписку с названием, стоимостью, периодичностью, категорией, датой платежа и напоминанием.
3. Пользователь ищет, фильтрует и сортирует подписки, чтобы быстро найти лишние расходы.
4. Пользователь анализирует месячные расходы, ближайшие платежи и категории с превышением лимита.

## Этапы разработки

| Срок | Задача |

| До конца апреля | Angular scaffold, routing, базовые экраны, формы, документация анализа и UX |

| Вторая половина мая | Аналитика, фильтры, сортировка, адаптивность, a11y, тесты |

## Риски

1. Нехватка времени на полный набор критериев — снижать риск короткими блоками и коммитами после каждой готовой части.
2. Ошибки совместимости Angular 21 и Taiga UI 4 — фиксировать версии в `package-lock.json` и регулярно запускать `npm run build`.
3. Недостаточное покрытие тестами — вынести расчеты в чистые функции и покрыть их Jest, а критичные пользовательские сценарии Playwright.
4. Формальное использование Taiga UI — применять компоненты в навигации, кнопках, уведомлениях, loader, badges и progress.
5. Проблемы деплоя — держать production build зеленым и использовать Vercel workflow с documented secrets.
